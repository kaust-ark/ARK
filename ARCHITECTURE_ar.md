# بنية ARK

## مبادئ التصميم

**الفكرة الأساسية**: ثق بحكم الذكاء الاصطناعي؛ الكود يتولى التنفيذ والحواجز الوقائية فقط.

- **قاعدة البيانات كمصدر وحيد للحقيقة** &mdash; إعدادات وحالة المشروع في SQLite؛ YAML للحالة التشغيلية للوكلاء فقط
- **عزل لكل مشروع** &mdash; كل مشروع يحصل على بيئة conda خاصة، HOME معزول، و `PYTHONNOUSERSITE=1`
- **المهارات بدلاً من القواعد المُبرمجة** &mdash; مجموعات تعليمات نمطية (skills) تُحمّل أثناء التشغيل لفرض أفضل الممارسات

## نظرة عامة على خط الأنابيب

يمرّ ARK بثلاث مراحل متتابعة:

### مرحلة Research (خط أنابيب من ٥ خطوات)

| الخطوة | الوكيل/الأداة | ما يحدث |
|:-------|:--------------|:--------|
| ٠ | — | **الإعداد**: تجهيز بيئة conda للمشروع (نسخ ark-base؛ عملية مثالية) |
| ١ | الباحث | **تحليل المقترح**: قراءة PDF المرفوع أو الفكرة → كتابة `idea.md`؛ إنتاج استعلام Deep Research؛ استخراج عنوان الورقة |
| ٢ | Gemini | **Deep Research**: مسح أدبي → `deep_research.md`؛ إرسال PDF عبر Telegram |
| ٣ | الباحث | **التخصيص**: توليد `project_context.md`؛ تخصيص قوالب الوكلاء للمشروع؛ اختيار المهارات (٠–٥) |
| ٤ | — | **التمهيد**: تثبيت المهارات المدمجة؛ تمهيد الاستشهادات → `references.bib` |

### حلقة المراجعة

كل تكرار يمرّ بـ ٥ خطوات: تجميع → مراجعة → تخطيط → تنفيذ → تحقّق.

المخطط يُخرج خطط عمل منظّمة بصيغة YAML:

```yaml
actions:
  - agent: experimenter
    task: "تشغيل تجربة التحقق من الحيرة"
    priority: 1
  - agent: writer
    task: "تحديث القسم ٤.٢"
    priority: 2
```

## المكونات الأساسية

### ١. نظام الذاكرة (`memory.py`)

يتتبع الدرجات، يكتشف الركود، ويمنع الإخفاقات المتكررة:

```python
class SimpleMemory:
    scores: List[float]       # سجل الدرجات (آخر ٢٠)
    best_score: float         # أعلى درجة تاريخية
    stagnation_count: int     # عدد مرات الركود المتتالية

    def record_score(score)   # تسجيل درجة
    def is_stagnating()       # كشف الركود
    def get_context()         # الحصول على السياق (مرساة الهدف + اتجاه الدرجات)
```

ميزات إضافية:
- **تتبع المشكلات**: إزالة التكرار بناءً على المحتوى — يحصي عدد مرات ظهور كل مشكلة عبر التكرارات
- **التحقق من الإصلاح**: يتأكد أن الإصلاحات المُجرّبة حلّت المشكلة فعلاً
- **تصعيد الاستراتيجية**: يحظر الأساليب غير الفعالة تلقائياً ويقترح بدائل
- **التصحيح الفوقي**: يُشغّل التشخيص عند توقف النظام

### ٢. مرساة الهدف

كل استدعاء لوكيل يتضمن "مرساة هدف" ثابتة تصف الأهداف الجوهرية للمشروع. هذا يمنع الوكلاء من الانحراف عن الموضوع عبر التكرارات المتعددة.

### ٣. المُنسّق (`orchestrator.py`)

يعتمد المُنسّق على تصميم قائم على المزيج (mixin) لتجميع الوظائف المتخصصة:

```python
class Orchestrator(AgentMixin, CompilerMixin, ExecutionMixin, PipelineMixin):
    # AgentMixin: استدعاء الوكلاء وتتبع التكلفة
    # CompilerMixin: تجميع LaTeX وإدارة ملفات PDF
    # ExecutionMixin: حقن المهارات وتنفيذ الأوامر
    # PipelineMixin: حلقات البحث والتطوير والمراجعة رفيعة المستوى
```

- **التوزيع** &mdash; يوزّع على المرحلة الصحيحة بناءً على وضع المشروع الحالي.
- **المزامنة** &mdash; يزامن الحالة والدرجات والتقدم مع قاعدة بيانات SQLite بعد كل خطوة.
- **التعامل** &mdash; يتعامل مع اتصالات Telegram ثنائية الاتجاه وقرارات التدخل البشري (HITL).

### ٤. نظام المهارات (`skills/`)

مجموعات تعليمات نمطية تُحمّل أثناء التشغيل لتوجيه سلوك الوكيل:

| المهارة | الغرض |
|:--------|:-------|
| **research-integrity** | مضاد للمحاكاة: الوكلاء يجب أن يُجروا تجارب حقيقية |
| **human-intervention** | بروتوكول التصعيد عبر Telegram للمعوقات |
| **env-isolation** | حدود بيئة لكل مشروع وأمنها |
| **figure-integrity** | التحقق من تطابق الأشكال مع البيانات التجريبية الفعلية |
| **page-adjustment** | التحكم في كثافة المحتوى للالتزام بحدود صفحات المؤتمر |

تُثبّت المهارات تلقائياً أثناء مرحلة التمهيد لخط الأنابيب (مرحلة البحث الخطوة ٤).

### ٥. عزل البيئات (`website/dashboard/jobs.py`)

يحصل كل مشروع على بيئة conda معزولة:

- `provision_project_env()` يستنسخ البيئة الأساسية إلى `<project>/.env/`
- `project_env_ready()` يتحقق من وجود البيئة
- المُنسّق يعمل بـ `HOME=<project_dir>` و `PYTHONNOUSERSITE=1`
- كل من CLI (`ark run`) ولوحة التحكم يكتشفان ويستخدمان بيئة المشروع تلقائياً.

### ٦. خلفيات الحوسبة (`ark/compute/`)

يدعم ARK خلفيات حوسبة متعددة لتشغيل التجارب:

- **Local**: تشغيل التجارب مباشرة على الجهاز المضيف.
- **Slurm**: إرسال المهام إلى مجموعات HPC باستخدام `sbatch`.
- **Cloud**: توفير مثيلات على **AWS** أو **GCP** أو **Azure**.
- **Custom**: خلفية قابلة للتوسيع للبيئات المتخصصة.

تتعامل الخلفيات السحابية مع الدورة الكاملة: التوفير، نقل الكود (rsync)، الإعداد، التنفيذ، جمع النتائج، والإنهاء.

### ٧. توليد الأشكال بالذكاء الاصطناعي (`ark/nano_banana.py`)

**Nano Banana** هو نظام يعمل بمحرك Gemini لتوليد أشكال علمية عالية الجودة:

- **المخطط**: يصمم مواصفات بصرية مفصلة بناءً على سياق الورقة.
- **المنسق**: يحسن المواصفات لتناسب جماليات النشر الأكاديمي.
- **المصور**: يولد الصورة باستخدام نماذج Gemini لتوليد الصور.
- **الناقد**: يقيم الشكل ويقدم ملاحظات للتحسين التكراري.

## قائمة الوكلاء (٦ وكلاء)

| الوكيل | الدور |
|:-------|:------|
| الباحث | يحلل المقترح → `idea.md`؛ مسح أدبي؛ يخصص موجّهات الوكلاء ويختار المهارات |
| المراجع | يراجع ويقيم الورقة؛ يتحقق من توافق التجارب مع المقترح |
| المخطط | يحلل المشكلات، يولد خطة عمل (وضعي الورقة والتطوير)؛ يتحقق من توافق التجارب |
| الكاتب | يكتب/يراجع أقسام الورقة باستشهادات موثقة عبر DBLP |
| المجرب | يصمم ويشغل ويحلل التجارب؛ يدعم خلفيات Slurm والسحابية |
| المبرمج | ينفذ تغييرات الكود (وضع التطوير) |

## مُهمل / محذوف

- `events.py` — نظام مدفوع بالأحداث (استُبدل بقرارات المخطط)
- تتبع ذاكرة معقد (issues, effective_actions, failed_attempts) — تم تبسيطه
- وكيل `initializer` — اندُمج في `researcher` (خطوة تحليل المقترح)
- وكيل `visualizer` — محذوف (كود ميت، لم يُستدعَ في خط الأنابيب)
- وكيل `meta_debugger` — محذوف (يُشخّص لكن لا يتصرف؛ استُبدل بكشف الركود في خط الأنابيب)
- وحدة Python `ark/webapp/` — نُقلت إلى `website/dashboard/`
